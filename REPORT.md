## А хэсэг — API Design

### А.1 Муу API шинжилгээ

Өгөгдсөн `usr_mgr` классаас дараах 6 дизайны алдааг илрүүллээ:

**1. Мэдээллийн далдлалтын зарчим зөрчигдсөн**  
`db_conn` болон `users_arr` талбарууд `public` байна. Доторх хэрэгжилтийн
мэдээлэл гадагш гарч байгаа нь хэрэглэгчийн кодыг хэрэгжилтээс хамааралтай
болгодог. Засвар: `private` болгох.

**2. Boolean flag abuse**  
`do_user_op(obj, flag, timeout)` функц нэг параметрийн утгаар 4 өөр үйлдэл
(create/update/delete/restore) хийнэ. Энэ нь функцийн зорилгыг тодорхойгүй
болгож, тестлэхэд хүндрэл учруулна. Засвар: 4 тусдаа метод болгох.

**3. Нэршлийн стандарт зөрчигдсөн**  
`usr_mgr`, `get_u`, `find` зэрэг нэрс нь утгагүй товчлол юм. TypeScript-д
camelCase болон PascalCase стандарт ашиглах ёстой. Засвар: `UserManager`,
`getUserById`, `searchUsers` болгох.

**4. Ойлголтын хэт их жин (Overloaded parameter)**  
`get_u(id_or_email, flag)` нэг параметрт хоёр өөр төрлийн утга хүлээж авна.
Энэ нь API-г ойлгоход хүндрэл учруулна. Засвар: `getUserById`, `getUserByEmail`
гэж тусдаа метод болгох.

**5. Алдааны буруу буцаалт**  
Функц `'ERR_404'` string буцаана. Алдааг string-ээр дамжуулах нь caller-д
алдааг шалгахад хүндрэл учруулж, алдагдах боломжтой. Засвар: Exception
throw хийх.

**6. Type safety алдагдсан**  
`any` төрлийг хэт их ашигласан. Энэ нь TypeScript-ийн ашиг тусыг алдагдуулна.
Засвар: `User`, `CreateUserInput`, `UpdateUserInput` интерфейс тодорхойлох.

### А.1 Сайжруулсан API

`UserManager` классыг дараах зарчмуудыг баримтлан бичлээ:

- **Private талбарууд**: `users: Map<string, User>`, `emailIndex: Map<string, string>`
- **Тодорхой методууд**: `createUser`, `getUserById`, `getUserByEmail`, `updateUser`, `deleteUser`, `restoreUser`, `searchUsers`
- **Exception**: `UserNotFoundException`, `UserAlreadyExistsException`, `InvalidParameterException`
- **TSDoc**: Бүх нийтийн метод баримтжуулагдсан

### А.2 Cache Library

#### Дизайны шийдвэр

**Factory паттерн** сонгосон шалтгаан:
- Хэрэглэгч зөвхөн `Cache` интерфейс болон `CacheFactory`-г мэдэх хэрэгтэй
- Concrete класс (`LRUCache`, `LFUCache`, `TTLCache`) гадагш гарахгүй
- Ирээдүйд шинэ хэрэгжилт нэмэхэд хэрэглэгчийн код өөрчлөгдөхгүй

#### Гурван хэрэгжилтийн харьцуулалт

| Шинж | LRU | LFU | TTL |
|------|-----|-----|-----|
| Устгах зарчим | Хамгийн удаан ашиглагдаагүй | Хамгийн бага давтамжтай | Хугацаа дууссан |
| Хэрэглэх тохиолдол | Ерөнхий кэш | Давтамжтай хандалт | Session, token |
| get/set цаг | O(1) | O(1) | O(1) |
| Нэмэлт санах ой | Doubly linked list | Frequency map | Expiry timestamp |

#### Unit тест үр дүн

- LRU: 5/5 тест давсан
- LFU: 5/5 тест давсан
- TTL: 5/5 тест давсан
- **Нийт: 15/15**

## Б хэсэг — REST API

### Архитектур
Client → Express.js Router → Handler → In-Memory DB
↓
Error Middleware (RFC 7807)

### Хэрэгжүүлсэн бизнесийн дүрмүүд

- Нэг гишүүн ≤ 5 ном зээлж болно → 409 Conflict
- Зээлийн хугацаа 14 хоног → dueDate автоматаар тооцогдоно
- Зээлийг нэг удаа сунгах → renewed flag ашиглан хянана
- Буцаагдсан зээлийг сунгаж болохгүй → 409 Conflict

### Postman тест үр дүн

- Нийт тест: 19
- Давсан: **19/19**
- Алдаа: 0

## Дүгнэлт

Энэхүү бие даалтаар API дизайны үндсэн зарчмуудыг практикаар судаллаа.
Мэдээллийн далдлалт, тодорхой нэршил, exception ашиглах зарчмууд нь
кодын чанар, засвар үйлчилгээ, тестлэх боломжид шууд нөлөөлдөг болохыг
ойлгосон. Factory паттерн нь сангийн дизайнд хэрэглэгчийн кодыг хэрэгжилтээс
тусгаарлах хамгийн үр дүнтэй арга болохыг харуулсан.
